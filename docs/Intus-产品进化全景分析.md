# Intus 产品进化全景分析

> 分析时间：2026-04-20 | 基于代码库全量扫描 | v2 修订版

---

## 一、产品现状概览

Intus 是一个 AI 驱动的智能访谈与洞察平台，核心能力链路：

**访谈设计 → AI 对话采集 → 结构化报告 → 方案页展示 → 分享协作**

### 当前已具备的能力

| 模块 | 能力 | 成熟度 |
|------|------|--------|
| 访谈引擎 | 多轮 AI 对话、场景模板、访谈脚本管理、下一题预检干预 | ★★★★ |
| 报告生成 | 结构化报告、Markdown 渲染、快照机制、降级链路 | ★★★★ |
| 方案页 | 报告绑定、方案展示、公开分享、旧报告 Markdown 回退兼容 | ★★★☆ |
| 用户体系 | 手机号/邮箱/微信登录、实例隔离、License 管理、账号合并 | ★★★☆ |
| 管理后台 | 用户管理、配置中心、归属迁移（preview/apply/rollback）、License 批量操作 | ★★★☆ |
| 数据管理 | S3 兼容对象存储、历史同步、云端导入/回滚 | ★★★ |
| 可观测性 | Harness 6 阶段自检、Browser Smoke、17 个 Evaluator 场景、稳定性 10 轮全通过 | ★★★★★ |

### 技术栈

- 后端：Python (Flask) — `web/server.py` 为主入口，`web/server_modules/` 按领域拆分
- 前端：Vanilla JS + 模块化 — `web/app.js` + `web/app_modules/`
- 数据库：PostgreSQL（云端生产）/ SQLite（本地开发），通过 `db_compat.py` 自动适配
- 文件存储：S3 兼容对象存储（AWS S3 / Cloudflare R2 / MinIO）
- AI：多模型接入（OpenAI/Claude），通过 `ai_service` 抽象层调用
- 部署：支持本地 / 云端双模式，环境配置通过 `.env.local` / `.env.cloud` / `.env.production` 区分

### 产品阶段判断

项目已完成 Harness 六阶段迭代，稳定性测试 S0-S6 全部通过，8 个高风险 task 均有 Sprint Contract 覆盖。当前处于 **"功能完整 → 产品化"的转折点**，heartbeat 指向"待新阶段计划"。

## 二、六大进化方向

### 方向一：体验层 — 从"能用"到"好用"

**现状痛点：**
- 报告生成是同步阻塞的，用户需要等待完整生成后才能看到内容
- 访谈过程缺少进度反馈，长访谈时用户不确定系统是否在工作
- 报告以静态 Markdown 呈现，缺少交互式探索能力

| 优先级 | 改进项 | 预期效果 | 改动范围 |
|--------|--------|----------|----------|
| P0 | 流式输出（SSE/WebSocket） | 报告边生成边展示，等待感降低 80% | `report_service` + 前端渲染层 |
| P0 | 访谈进度条 | 用户清楚当前进度和预计剩余时间 | 访谈引擎 + 前端 UI |
| P1 | 交互式报告阅读 | 点击章节展开/折叠，关键数据可钻取 | `app_modules/report` |
| P1 | 报告对比视图 | 同一主题多次访谈结果并排对比 | 新增模块 |
| P2 | 暗色模式 | 长时间阅读体验优化 | CSS 主题层 |

**流式输出技术路径：**
```
当前：用户点击生成 → 等待 30-120s → 一次性返回完整报告
目标：用户点击生成 → 立即看到骨架 → 逐段填充 → 完成提示
```
核心改动在 `web/server_modules/report_service` 中将同步调用改为 SSE 流，前端用 EventSource 逐段渲染。

---

### 方向二：协作层 — 从"个人工具"到"团队平台"

**现状痛点：**
- 当前是单用户模式，没有团队/组织概念
- 报告分享只有公开链接，缺少团队内部协作
- 没有评论、标注、任务分配等协作功能

| 优先级 | 改进项 | 预期效果 |
|--------|--------|----------|
| P0 | 团队空间（Workspace） | 团队成员共享访谈和报告，基于现有 `instance_scope_key` 扩展 |
| P0 | 角色权限体系 | 管理员/编辑者/查看者三级权限 |
| P1 | 报告评论与标注 | 团队成员可在报告段落上评论讨论 |
| P1 | @提及与通知 | 评论中 @同事，触发站内/邮件通知 |
| P2 | 多人协同访谈 | 多个访谈者同时参与一场访谈，各自视角汇总 |
| P2 | 访谈任务分配 | 管理者创建访谈任务，分配给团队成员执行 |

**团队空间实现路径：**
现有 `instance_scope_key` 已做实例级隔离，扩展关键步骤：
1. 新增 `team` 表（team_id, name, owner_user_id, instance_scope_key）
2. 新增 `team_member` 表（team_id, user_id, role）
3. 所有查询加入 team_id 过滤条件
4. 前端增加团队切换器

---

### 方向三：智能层 — 从"AI 执行"到"AI 洞察"

**现状痛点：**
- AI 主要用于执行访谈对话和生成报告文本
- 缺少跨访谈的知识积累和趋势分析
- 报告质量依赖 prompt 设计，缺少自动优化机制

| 优先级 | 改进项 | 预期效果 |
|--------|--------|----------|
| P0 | 行动计划生成 | 报告末尾自动生成可执行的行动建议，带优先级和负责人 |
| P1 | 知识库沉淀 | 多次访谈的洞察自动归类、去重、形成组织知识图谱 |
| P1 | 质量自学习 | 基于用户对报告的编辑行为，自动优化生成 prompt |
| P2 | 趋势分析 | 同一主题多次访谈的变化趋势可视化 |
| P2 | 智能推荐 | 基于历史访谈推荐下一步应该关注的问题 |
| P2 | 多模态输入 | 支持语音访谈、图片/文档上传作为访谈素材 |

### 方向四：商业化层 — 从"免费工具"到"付费产品"

| 优先级 | 改进项 | 预期效果 |
|--------|--------|----------|
| P0 | 分层定价（Free/Pro/Enterprise） | 明确的商业模式，Free 版限制访谈次数和报告长度 |
| P0 | 用量仪表盘 | 用户清楚自己的使用量和剩余配额 |
| P1 | 增长钩子 | 邀请好友获得额外配额、分享报告带品牌水印 |
| P1 | 企业 SSO | SAML/OIDC 接入，企业客户的硬性需求 |
| P2 | 计费 API | 按调用量计费的 API 接入模式 |

**定价模型建议：**

| 层级 | 价格 | 访谈次数 | 报告长度 | 团队成员 | 特色功能 |
|------|------|----------|----------|----------|----------|
| Free | ¥0 | 5次/月 | 标准 | 1人 | 基础访谈 + 报告 |
| Pro | ¥99/月 | 50次/月 | 深度 | 5人 | 团队空间 + 知识库 |
| Enterprise | 定制 | 不限 | 不限 | 不限 | SSO + API + 私有部署 |

### 方向五：生态层 — 从"封闭系统"到"开放平台"

| 优先级 | 改进项 | 预期效果 |
|--------|--------|----------|
| P0 | 开放 API v1 | RESTful API 覆盖访谈创建、报告获取、方案页嵌入 |
| P0 | 场景模板市场 | 用户可创建、发布、订阅访谈场景模板 |
| P1 | Webhook 事件通知 | 访谈完成、报告生成等事件推送到客户系统 |
| P1 | 嵌入式 SDK | iframe/Web Component，让客户将访谈嵌入自己的产品 |
| P2 | 插件系统 | 允许开发者扩展报告分析维度、接入自定义数据源 |

**开放 API 核心端点：**
```
POST   /api/v1/interviews          创建访谈
GET    /api/v1/interviews/:id      获取访谈详情
POST   /api/v1/interviews/:id/run  执行访谈
GET    /api/v1/reports/:id         获取报告
POST   /api/v1/reports/:id/export  导出报告
GET    /api/v1/solutions/:id       获取方案页
POST   /api/v1/webhooks            注册 Webhook
```

**现状：** 数据库已迁移至 PostgreSQL，文件存储已接入 S3 兼容服务，基础设施层已具备云原生基础。

| P2 | 国际化（i18n） | 当前仅中文，扩展英文等多语言支持 |

## 三、具体实施计划

### Phase 1：体验闭环 + 商业化基础 + 引擎解耦（0-3个月）

**目标：** 让产品从"能用"变成"愿意付费用"，同时为 IaaS 埋下技术基础

> 设计原则：Phase 1 的每个 Sprint 都同时服务"自用产品"和"未来平台化"两个目标。
> 不做额外的平台化工作，而是在做产品功能时，用"可复用"的方式做。

#### Sprint 1（第 1-2 周）：流式输出

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | 后端：`report_service` 新增 SSE 端点 `/api/report/stream` | SSE 接口 | curl 能收到逐段事件 |
| 2 | 后端：AI 调用层改为流式（`ai_service` 的 stream=True） | 流式适配层 | 单元测试通过 |
| 3 | 前端：`app_modules/report` 新增 EventSource 渲染器 | 流式渲染组件 | 报告逐段出现，无闪烁 |
| 4 | 前端：加载骨架屏 + 进度指示器 | UI 组件 | 用户点击后立即有反馈 |
| 5 | 兼容：旧报告仍走同步接口，新报告默认流式 | 特性开关 | 两种模式均可正常工作 |
| 6 | Browser Smoke 补充流式场景 | 测试用例 | CI 通过 |

**IaaS 伏笔：** SSE 端点设计为通用协议（event type + payload JSON），未来嵌入式 SDK 直接复用。

#### Sprint 2（第 3-4 周）：团队空间

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | 数据库：新增 `team`、`team_member` 表，PostgreSQL migration | DDL 脚本 | 迁移可回滚 |
| 2 | 后端：团队 CRUD API + 邀请/移除成员 | REST 接口 | API 测试通过 |
| 3 | 后端：所有查询加入 team_id 过滤（在现有 scope_key 基础上） | 查询层改造 | 跨团队数据不可见 |
| 4 | 前端：团队切换器 + 成员管理页 | UI 页面 | 可创建团队、邀请成员 |
| 5 | 权限：管理员/编辑者/查看者三级权限校验 | 中间件 | 越权操作被拦截 |

**IaaS 伏笔：** 团队 = 未来的"租户"。权限中间件设计为 `scope(team_id, role)` 通用模式，未来 API Key 也走同一套。

#### Sprint 3（第 5-6 周）：行动计划 + 分层定价

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | 报告生成 prompt 增加"行动计划"章节 | Prompt 模板 | 报告末尾有可执行建议 |
| 2 | 前端：行动计划卡片组件（优先级、负责人、截止日期） | UI 组件 | 可交互编辑 |
| 3 | License 模块扩展：Free/Pro/Enterprise 三级配额 | 配额引擎 | 超限时提示升级 |
| 4 | 用量仪表盘：当月访谈次数、报告数、存储用量 | 仪表盘页面 | 数据准确 |

**IaaS 伏笔：** 配额引擎设计为 `quota(scope_id, resource_type, limit)` 通用模型，未来按次计费直接复用。

#### Sprint 4（第 7-8 周）：开放 API v1 + 引擎解耦（关键 Sprint）

> 这是整个路线中最关键的 Sprint — 既交付开放 API，又完成引擎解耦，为 IaaS 打通技术通道。

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | API Key 管理：生成、吊销、权限范围 | 密钥管理模块 | 可在设置页管理 |
| 2 | 核心端点实现（访谈创建/执行/报告获取） | REST API | Postman 集合测试通过 |
| 3 | 请求频率限制 + 用量计量 | 中间件 | 超限返回 429 |
| 4 | API 文档（OpenAPI/Swagger） | 文档站 | 开发者可自助接入 |
| 5 | **引擎解耦：维度配置从前端移到后端 scenario_config** | 配置驱动 | 新场景只需 JSON，无需改前端代码 |
| 6 | **引擎解耦：覆盖度阈值、追问规则参数化** | 可配置引擎 | 不同场景可配不同策略 |

**为什么放在这个 Sprint：** 开放 API 本身就要求"通过 API 创建访谈时传入场景配置"，这天然倒逼引擎解耦。两件事一起做，成本最低。

**解耦后的 API 调用示例：**
```json
POST /api/v1/interviews
{
  "scenario": {
    "dimensions": [
      {"id": "satisfaction", "name": "满意度", "min_questions": 2, "max_questions": 4},
      {"id": "pain_points", "name": "痛点", "min_questions": 3, "max_questions": 5}
    ],
    "completion_threshold": 80,
    "probing_strategy": "moderate",
    "output_format": "structured_json"
  }
}
```

这个 API 既服务于"开放 API v1"的需求，又是未来 IaaS 的核心接口。

#### Sprint 5（第 9-10 周）：多租户正式化 + 增长钩子

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | 租户级配置隔离（品牌、模型参数、功能开关） | 配置层 | 不同租户配置独立 |
| 2 | 租户级数据完全隔离审计 | 安全测试 | 跨租户查询零结果 |
| 3 | 邀请注册奖励机制 | 增长模块 | 邀请链接可追踪 |
| 4 | 分享报告品牌水印（Free 版） | 水印组件 | Pro 版无水印 |

**IaaS 伏笔：** 租户配置隔离 = 未来每个 IaaS 客户的独立配置空间。品牌定制能力直接服务于白标嵌入。

#### Sprint 6（第 11-12 周）：集成测试 + 发布 + IaaS 验证

| 步骤 | 具体任务 | 产出物 | 验收标准 |
|------|----------|--------|----------|
| 1 | 全链路集成测试（登录→团队→访谈→报告→分享→API） | 测试套件 | 端到端通过 |
| 2 | 性能基准测试（并发访谈、流式报告） | 性能报告 | P95 延迟达标 |
| 3 | 文档更新 + 迁移指南 | 用户文档 | 老用户可平滑升级 |
| 4 | **IaaS 概念验证：用开放 API 搭建一个"客户反馈收集"的 demo** | Demo 页面 | 证明引擎可支持非访谈场景 |
| 5 | 灰度发布 + 监控告警 | 发布计划 | 回滚方案就绪 |

**Phase 1 结束时的状态：** 产品本身完成体验升级和商业化基础，同时引擎已解耦、API 已就绪、有一个 IaaS demo 验证了可行性。

### Phase 2：深度价值 + IaaS 平台化（3-6个月）

> Phase 2 的核心转变：从"给自己做功能"变成"同时给自己和第三方做能力"。
> 每个 Sprint 的产出都有两个视角：自用产品视角 + IaaS 平台视角。

| Sprint | 自用产品交付 | IaaS 平台交付 | 关键里程碑 |
|--------|-------------|---------------|------------|
| S7-S8 | 知识库沉淀：知识抽取 + 结构化存储 + 搜索 | 访谈结果结构化输出 API（JSON/CSV/Webhook） | 用户可搜索历史洞察；API 客户可拿到结构化数据 |
| S9-S10 | 交互式报告：折叠/展开 + 数据钻取 + 对比 | **场景编辑器 v1**：可视化定义维度、问题策略、输出格式 | 报告可交互探索；非技术用户可自助创建访谈场景 |
| S11-S12 | 场景模板市场：创建/发布/订阅 + 审核 | **嵌入式 SDK v1**：iframe + postMessage 通信协议 | 首批 20 个社区模板；首个第三方产品嵌入 demo |
| S13-S14 | 企业 SSO（SAML/OIDC）+ Webhook 事件通知 | IaaS 客户管理后台：用量统计、计费、API Key 管理 | 首个企业客户接入；首个 IaaS 付费客户 |
| S15-S16 | 质量自学习：编辑行为采集 + prompt 自动调优 | **按次计费网关**：API 调用计量 + 账单生成 | 报告满意度可量化；IaaS 商业模式跑通 |

**Phase 2 的关键产出 — 场景编辑器：**

```
场景编辑器界面：
┌─────────────────────────────────────────────┐
│  场景名称：[客户满意度调查]                      │
│                                             │
│  维度配置：                                    │
│  ┌─ 整体满意度  [2-3题]  追问策略：[温和]  ──┐  │
│  ├─ 功能痛点    [3-5题]  追问策略：[深入]  ──┤  │
│  ├─ 改进建议    [2-3题]  追问策略：[开放]  ──┤  │
│  └─ 竞品对比    [1-2题]  追问策略：[温和]  ──┘  │
│                                             │
│  完成阈值：[80%]  总题数上限：[12]              │
│  输出格式：[结构化JSON] [Markdown报告]          │
│                                             │
│  [预览访谈]  [保存为模板]  [发布到市场]          │
└─────────────────────────────────────────────┘
```

这个编辑器同时服务于：
- 自用用户：创建自定义访谈场景，发布到模板市场
- IaaS 客户：定义自己产品中嵌入的访谈流程

### Phase 3：IaaS 规模化 + 差异化壁垒（6-12个月）

> Phase 3 的核心目标：让 IaaS 从"能用"变成"好用且有生态"。

| Sprint | 核心交付 | IaaS 维度 | 关键里程碑 |
|--------|----------|-----------|------------|
| S17-S20 | 多人协同访谈：多视角采集 + 观点汇总 | **行业方案包**：与 HR/医疗/教育伙伴共建首批 3 个垂直方案 | 团队访谈可用；首个行业方案包上线 |
| S21-S24 | 嵌入式 SDK v2：Web Component + 主题定制 + 事件回调 | **SDK 开发者文档站** + 沙箱环境 | 第三方可自助接入，无需对接支持 |

## 四、最高优先级 TOP 3

如果资源有限只能做三件事：

1. **流式输出** — 当前等待体验是最大的用户流失点，技术改动集中在 `report_service` 和前端渲染层，投入产出比最高。SSE 协议设计为通用格式，未来嵌入式 SDK 直接复用。

2. **开放 API + 引擎解耦（合并做）** — 开放 API 天然倒逼引擎解耦：API 调用者需要传入场景配置，这要求维度定义、追问策略、完成阈值都可参数化。一个 Sprint 同时交付 API 和引擎通用化，是 IaaS 的技术起点。

3. **团队空间** — 从个人工具变成团队工具是 B 端付费的前提。团队 = 未来的租户，权限模型 = 未来 API Key 的权限模型，配额引擎 = 未来按次计费的基础。

## 五、竞争定位思考

Intus 的独特价值在于 **"AI 驱动的结构化访谈"** 这个细分赛道：

- 对比通用 AI 对话工具（ChatGPT/Claude）：Intus 提供的是 **场景化、结构化、可沉淀** 的访谈体验，而非开放式对话
- 对比传统调研工具（问卷星/SurveyMonkey）：Intus 用 AI 实现了 **动态追问、深度挖掘**，远超固定问卷的信息密度
- 对比用户研究平台（UserTesting/Maze）：Intus 的 **自动化程度更高**，不依赖真人主持

护城河构建的关键：**场景模板生态 + 组织知识积累 + 开放 API 集成网络**

## 六、精益求精 — 深层建议

以下是从代码库深度扫描中发现的、前面未覆盖的进化机会：

### 6.1 用户旅程断点修复

当前用户旅程 `登录 → License 激活 → 创建访谈 → 逐题推进 → 生成报告 → 方案页 → 分享` 存在几个隐性断点：

| 断点 | 问题 | 建议 |
|------|------|------|
| License 过期中断 | 用户访谈进行中 License 过期，缺少优雅降级 | 增加"宽限期"机制，允许完成当前访谈 |
| 报告生成队列满载 | 前端加载态与后端队列状态不同步 | 增加队列位置提示 + 预估等待时间 |
| 快照版本不一致 | 方案页消费报告快照，旧报告 Markdown fallback 与新快照结构的兼容边界模糊 | 统一版本协议，增加快照版本号校验 |
| 降级后的业务连续性 | 微信登录失败、视觉超时、存储归档失败都有失败注入测试，但缺少"降级后用户能否继续完整流程"的验证 | 补充降级后的端到端 smoke test |

### 6.2 数据一致性加固

已有 PostgreSQL + S3 的云原生基础，但以下场景的一致性保障可以加强：

| 场景 | 风险 | 建议 |
|------|------|------|
| 并发访谈 + 报告生成 + 分享创建 | 缺乏全链路并发压力测试 | 补充 locust/k6 压测场景 |
| 账号合并后的历史数据 | 分享映射、导出资产的跨实例一致性未充分验证 | 补充合并后的数据完整性校验 |
| 对象存储并发一致性 | `object_storage_history.py` 的历史补齐仅在迁移场景触发 | 增加日常写入的并发监控 |

### 6.3 可观测性从"自检"到"运营"

当前 Harness 体系非常完善（6 阶段、17 个 evaluator 场景、稳定性 10 轮全通过），但偏向开发态自检。建议向运营态扩展：

| 维度 | 当前 | 建议 |
|------|------|------|
| 用户行为分析 | 无 | 接入埋点（访谈完成率、报告阅读深度、分享转化率） |
| AI 质量监控 | 依赖 evaluator 场景 | 增加生产环境的报告质量评分（用户满意度 + AI 自评） |
| 成本归因 | 无 | 按租户/场景统计 AI 调用成本，为定价提供数据支撑 |
| 配置变更影响 | 手工 playbook | `admin_config_center` 的配置变更增加自动化影响验证 |

### 6.4 前端架构现代化

当前 Vanilla JS + 模块化的架构在功能简单时够用，但随着团队空间、交互式报告、知识库等复杂功能的加入，建议评估：

| 选项 | 优势 | 风险 | 建议时机 |
|------|------|------|----------|
| 维持现状 + 渐进增强 | 零迁移成本 | 复杂交互开发效率低 | Phase 1 |
| 引入轻量框架（Preact/Alpine.js） | 低迁移成本，渐进式 | 需要建立新的组件规范 | Phase 2 开始时 |
| 迁移到 React/Vue | 生态完善，长期效率高 | 迁移成本大，需要并行维护 | Phase 3 如果团队扩大 |

建议 Phase 1 维持现状，Phase 2 开始时引入 Preact 或 Alpine.js 做渐进增强，避免一次性大迁移。

### 6.5 安全加固清单

| 项目 | 当前状态 | 建议 |
|------|----------|------|
| API 认证 | 登录态 Cookie | 开放 API 需要独立的 API Key + HMAC 签名 |
| 速率限制 | 无全局限制 | 按 IP / 用户 / API Key 三级限流 |
| 数据加密 | 传输层 HTTPS | 敏感字段（访谈内容）增加应用层加密 |
| 审计日志 | 管理后台有操作日志 | 扩展到所有写操作，支持合规审计 |
| GDPR/数据删除 | 有删除功能 | 增加"彻底删除"（含 S3 对象 + 数据库记录 + 缓存） |

### 6.6 访谈即服务（Interview-as-a-Service）— 深度展开

> 这是本文档中最值得深入思考的方向。Intus 最独特的资产不是报告，而是结构化访谈引擎本身。

---

#### 核心洞察：你拥有的不只是一个产品，而是一个引擎

从代码库深度扫描来看，Intus 的访谈引擎已经具备了一套相当完整的通用能力：

| 引擎能力 | 代码位置 | 通用化潜力 |
|----------|----------|------------|
| 多轮对话控制 | `interview_runtime` | 任何需要结构化信息采集的场景都需要 |
| 维度覆盖度追踪 | coverage % 计算 | 可抽象为"信息完整度"指标 |
| 证据 Ledger 系统 | focus_slots + blocked_sections | 可抽象为"关键信息缺口检测" |
| 追问决策引擎 | eval_signals + blindspot 检测 | 可抽象为"深度挖掘策略" |
| 预检干预 | mid_interview_preflight | 可抽象为"质量门控" |
| 多 Lane 问题生成 | fast/full lane + hedge + fallback | 可抽象为"自适应问题策略" |
| 超时恢复 | watchdog timer + stall grace | 任何长对话都需要 |

这些能力组合在一起，构成了一个 **"结构化信息采集引擎"** — 它的价值远超"做访谈"这一个场景。

#### 为什么这是最大的机会

**从市场角度看：**

当前市场上有大量"AI 对话"产品，但几乎没有产品解决 **"如何用 AI 进行有目的、有结构、有质量保证的信息采集"** 这个问题。

- ChatGPT/Claude：开放式对话，没有结构，没有完整度追踪
- 问卷工具：有结构，但没有智能追问，信息密度低
- 表单工具：纯数据收集，没有对话能力
- 客服机器人：有对话，但目标是解决问题而非采集信息

Intus 的引擎恰好填补了这个空白：**有结构 + 有智能 + 有质量保证的信息采集**。

**从技术角度看：**

引擎的核心能力已经高度参数化：

```
可配置项：
├── 维度定义（dimensions）— 采集什么信息
├── 问题数量（min/max_questions）— 采集多深
├── 追问策略（eval_signals）— 什么时候深挖
├── 证据要求（focus_slots）— 什么信息必须拿到
├── 输出模式（light/full）— 轻量还是深度
├── 搜索行为（rule_only/default/aggressive）— 是否联网补充
└── 上下文预算（doc_budget）— 参考资料的丰富度
```

这意味着只要解决几个硬编码限制，引擎就能支持完全不同的行业场景。

#### 五个高价值落地场景

**场景一：客户成功 — 产品内嵌式用户反馈**

```
传统方式：发 NPS 问卷 → 用户填 1-10 分 → 得到一个数字，不知道为什么
IaaS 方式：产品内弹出 AI 访谈 → 动态追问痛点 → 自动生成结构化反馈报告
```

| 维度 | 配置 |
|------|------|
| 采集目标 | 使用满意度、功能痛点、改进建议 |
| 问题数量 | 3-5 题（轻量，不打扰用户） |
| 追问策略 | 仅在用户表达不满时深挖 |
| 输出格式 | 结构化 JSON（直接入客户成功系统） |
| 嵌入方式 | iframe / Web Component |

**场景二：HR — 员工体验与离职面谈**

```
传统方式：HR 手动面谈 → 主观记录 → 信息丢失
IaaS 方式：AI 结构化访谈 → 标准化维度 → 自动归类 → 趋势分析
```

| 维度 | 配置 |
|------|------|
| 采集目标 | 工作满意度、团队协作、职业发展、离职原因 |
| 问题数量 | 8-12 题（深度访谈） |
| 追问策略 | 对敏感话题温和追问，尊重边界 |
| 输出格式 | 脱敏报告（保护员工隐私）+ 聚合趋势 |
| 特殊要求 | 匿名模式、数据加密、合规审计 |

**场景三：医疗健康 — 症状预筛查**

```
传统方式：患者到院 → 排队 → 医生问诊 15 分钟 → 信息不完整
IaaS 方式：就诊前 AI 预筛查 → 结构化症状采集 → 医生拿到完整信息
```

| 维度 | 配置 |
|------|------|
| 采集目标 | 主诉症状、持续时间、严重程度、既往病史、用药情况 |
| 问题数量 | 5-8 题（患者耐心有限） |
| 追问策略 | 对危险信号（胸痛、呼吸困难）立即标记 |
| 输出格式 | 标准化病历摘要（对接 HIS 系统） |
| 特殊要求 | 医疗合规、数据本地化、紧急情况转人工 |

**场景四：教育培训 — 能力诊断与学习路径**

```
传统方式：做一套标准化测试 → 得到分数 → 不知道具体薄弱点
IaaS 方式：AI 对话式诊断 → 动态调整难度 → 精准定位知识缺口 → 推荐学习路径
```

**场景五：咨询/审计 — 标准化尽调**

```
传统方式：咨询顾问手动访谈 → 每人风格不同 → 信息质量参差
IaaS 方式：标准化访谈框架 → AI 保证覆盖度 → 自动生成尽调报告
```

#### 当前引擎的通用化障碍与解法

从代码扫描中识别出 5 个需要解决的硬编码限制：

| 障碍 | 现状 | 解法 | 工作量 |
|------|------|------|--------|
| 维度顺序前端硬编码 | `dimensionOrder` 写死在 `interview_runtime.js` | 改为从后端 scenario_config 动态获取 | 小 |
| 覆盖度阈值固定 | coverage >= 100 完成，>= 50 可跳过 | 参数化为 scenario_config 的一部分 | 小 |
| 追问规则与业务耦合 | eval_signals 的判断逻辑硬编码 | 抽象为可配置的规则引擎 | 中 |
| 报告模板与维度强绑定 | sections 与 dimensions 一一对应 | 引入模板 DSL，支持动态映射 | 中 |
| 前端状态机与流程耦合 | currentStep 0→1→2 线性流程 | 抽象为状态机配置，支持分支/条件 | 大 |

**关键判断：前三个障碍工作量小，解决后就能支持 80% 的新场景。**

#### 产品形态：三层架构

```
┌─────────────────────────────────────────────────┐
│              应用层（场景解决方案）                  │
│  用户研究  │  HR 访谈  │  症状筛查  │  能力诊断     │
├─────────────────────────────────────────────────┤
│              平台层（配置与管理）                    │
│  场景编辑器  │  模板市场  │  数据看板  │  集成管理    │
├─────────────────────────────────────────────────┤
│              引擎层（核心能力）                      │
│  对话控制  │  证据Ledger  │  追问策略  │  质量门控    │
└─────────────────────────────────────────────────┘
```

- **引擎层**：当前已有，需要解耦硬编码
- **平台层**：需要新建，核心是"场景编辑器"— 让非技术用户也能定义访谈场景
- **应用层**：基于平台层配置，针对不同行业的预置方案

#### 商业模式变化

从"卖产品"变成"卖能力"：

| 模式 | 收费方式 | 目标客户 | 预期 ARPU |
|------|----------|----------|-----------|
| SaaS 自用 | 按席位/月 | 中小企业 | ¥99-499/月 |
| 嵌入式 SDK | 按访谈次数 | SaaS 产品方 | ¥0.5-2/次 |
| 私有部署 | 年度授权 | 大企业/医疗/金融 | ¥10-50万/年 |
| 场景市场分成 | 模板销售抽成 | 行业专家/咨询公司 | 平台抽成 20-30% |

嵌入式 SDK 的按次计费模式最有想象空间 — 如果 10 个 SaaS 产品各自嵌入，每个每天产生 100 次访谈，就是日均 1000 次调用。

#### 实施路径：三步走

**第一步（Phase 1 末尾，+2 周）：引擎解耦**
- 维度配置从前端移到后端 scenario_config
- 覆盖度阈值参数化
- 追问规则可配置化
- 产出：引擎可以通过 JSON 配置支持新场景，无需改代码

**第二步（Phase 2 中期，4 周）：场景编辑器 + 嵌入式 SDK**
- 可视化场景编辑器（维度定义、问题策略、输出格式）
- iframe / Web Component 嵌入方案
- 按次计费的 API 网关
- 产出：第三方可以自助创建场景并嵌入自己的产品

**第三步（Phase 3，持续）：行业方案 + 模板市场**
- 与行业伙伴共建垂直场景（HR、医疗、教育）
- 模板市场上线，支持社区贡献
- 产出：形成生态飞轮

#### 这个方向的风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 通用化导致核心场景体验下降 | 中 | 高 | 保持"用户研究"作为旗舰场景，通用化不降级旗舰体验 |
| 行业场景的领域知识不足 | 高 | 中 | 与行业伙伴共建，而非自己做所有行业 |
| 嵌入式场景的安全合规要求高 | 中 | 高 | 医疗/金融场景优先支持私有部署 |
| 场景编辑器的易用性挑战 | 中 | 中 | 先做"模板 fork + 微调"，再做完全自定义 |
