# Intus 智能体定价方案

版本日期：2026-05-08

## 1. 结论

Intus 不应按「聊天次数」定价，而应按「完成一个可交付的需求洞察项目」定价。当前产品主链路是「发起访谈 -> 沉淀记录 -> 生成报告 -> 派生方案页 -> 导出与分享」，适合按工作流交付价值、可变计算成本和企业治理能力组合计费。

建议采用以下结构：

```text
基础平台费
+ included credits
+ 超额 credits
+ 高价值交付包
+ 企业治理费
```

首期建议直接上线 3 个自助档位和 2 类销售驱动套餐：

| 类型 | 建议 |
|---|---|
| 自助主档位 | 体验版、标准版、专业版 |
| 团队/企业档位 | 团队版、企业版 |
| 结果型收入 | 专业交付包、企业试点包 |
| 核心约束 | 禁止承诺 `unlimited`，所有高成本任务都走 credits 预算 |

## 2. 产品边界

### 2.1 工作单元

| 工作单元 | 定义 | 典型交付物 |
|---|---|---|
| 标准项目 | 标准访谈 + `balanced` 报告 + 正文导出 | Markdown、PDF、Word |
| 专业交付包 | 深度访谈 + `quality` 报告 + 方案页 + 分享 + 附录 + 演示稿 | 报告、方案页、分享链接、附录、PPT |
| 企业治理能力 | 多账号、管理员、License、配置中心、审计、实例隔离、对象存储 | 管理后台、审计记录、私有化/专属实例 |

### 2.2 替代工作流

Intus 替代的不是一次问答，而是以下人工流程：

- 需求访谈提纲设计。
- 访谈推进与追问。
- 访谈记录整理。
- 洞察报告生成。
- 方案页和汇报材料沉淀。
- 导出、分享、归档和权限控制。

### 2.3 失败后果

| 场景 | 风险等级 | 定价含义 |
|---|---|---|
| 体验报告生成失败 | 中 | 可重试、可降级，不适合高价结果承诺 |
| 标准交付失败 | 中高 | 需要预算重试、失败提示和导出兜底 |
| 专业方案页或演示稿失败 | 高 | 必须计入外部服务、人工复核和补偿成本 |
| 企业环境配置/迁移失败 | 高 | 不应放在自助 credits 内，必须销售驱动并保留审计 |

## 3. 当前能力映射

Intus 已有 `experience`、`standard`、`professional` 三档 License。商业定价应直接贴合这套能力，而不是另起一套 `Free / Pro / Enterprise` 口径。

| 能力 | 体验版 | 标准版 | 专业版 |
|---|---:|---:|---:|
| 快速访谈 | 支持 | 支持 | 支持 |
| 标准访谈 | 不支持 | 支持 | 支持 |
| 深度访谈 | 不支持 | 不支持 | 支持 |
| `balanced` 报告 | 支持 | 支持 | 支持 |
| `quality` 报告 | 不支持 | 不支持 | 支持 |
| Markdown / PDF / Word 导出 | 不支持 | 支持 | 支持 |
| 附录导出 | 不支持 | 不支持 | 支持 |
| 方案页查看 | 不支持 | 不支持 | 支持 |
| 方案页分享 | 不支持 | 不支持 | 支持 |
| 演示文稿生成 | 不支持 | 不支持 | 支持 |

## 4. 成本模型

### 4.1 成本公式

```text
direct run cost = model + tools + infra + APIs + storage
effective completion cost = total run cost / successful completions
floor price = effective completion cost / (1 - target gross margin)
```

目标毛利建议：

| 阶段 | 目标毛利 | 说明 |
|---|---:|---|
| 内测/试点 | 50%-60% | 允许人工介入，重点验证价值 |
| 自助 SaaS | 70%-80% | 标准化交付，必须控制重试和超额用量 |
| 企业交付 | 60%-75% | 包含实施、支持、治理和 SLA |

### 4.2 本地观测口径

当前本地运行指标显示：

| 指标 | 观测值 | 含义 |
|---|---:|---|
| 总调用成功率 | 约 85.2% | 自助链路可用，但需保留失败预算 |
| 报告相关成功率 | 约 68.7% | 专业交付不能只按 token 成本定价 |
| 平均响应时间 | 约 22 秒 | 需要限制并发、队列和重试 |
| 近期风险 | timeout、502、报告 draft/review 失败 | credits 需覆盖重试和降级成本 |

说明：以上来自本地 `data/metrics/api_metrics.json` 快照，只用于定价假设，不应直接作为线上 SLA 承诺。

### 4.3 单次交付成本假设

| 工作单元 | credits | 内部有效成本估算 | 75% 毛利地板价 |
|---|---:|---:|---:|
| 轻量体验报告 | 40-80 | ¥5-15 | ¥20-60 |
| 标准项目 | 180-240 | ¥25-40 | ¥100-160 |
| 专业交付包 | 600-900 | ¥100-180 | ¥400-720 |
| 人工复核交付 | 1,500-3,000 | ¥250-500 | ¥1,000-2,000 |

## 5. 价值模型

### 5.1 价值公式

```text
value ceiling =
旧工作流成本
x 质量系数
x 成功率
x 责任系数

用户侧成本倍率 =
旧工作流成本 / 有效完成成本
```

### 5.2 典型价值区间

| 场景 | 旧工作流成本 | Intus 有效成本 | 用户侧成本倍率 | 定价姿态 |
|---|---:|---:|---:|---|
| 标准需求访谈与报告 | ¥1,500-3,000 | ¥25-40 | 约 38x-120x | 自助订阅 + credits |
| 售前方案沉淀 | ¥5,000-15,000 | ¥100-180 | 约 28x-150x | 专业版 + 交付包 |
| 咨询式业务诊断 | ¥15,000-50,000 | ¥250-500 | 约 30x-200x | 销售驱动 + 结果型定价 |

结论：Intus 的机会不是低价工具，而是高价值交付工作流。自助档位负责获客和标准化使用，专业交付包负责捕获核心价值。

## 6. Credits 规则

### 6.1 基础定义

建议定义：

```text
1 credit = ¥0.10 内部直接成本预算
零售价格 = ¥0.50-¥1.00 / credit
```

实际账务中，credits 不应只等同于 token，而应覆盖模型、搜索、文件处理、外部演示稿服务、对象存储、队列、重试和人工复核准备金。

### 6.2 任务扣费建议

| 任务 | 建议 credits | 成本驱动 |
|---|---:|---|
| 简单问答/轻量追问 | 1-3 | tokens |
| 单文档摘要 | 3-10 | tokens、文件解析 |
| 快速访谈问题生成 | 5-15 | question lane、hedge |
| 标准访谈推进 | 10-30 | 上下文、追问、质量判断 |
| 标准报告生成 | 120-200 | draft、review、质量门 |
| `quality` 报告生成 | 250-450 | 更多轮次、更多 evidence、修复重试 |
| 方案页生成/刷新 | 80-180 | 报告快照、结构化渲染 |
| 附录导出 | 30-80 | 文件生成、归档 |
| 演示稿生成 | 300-600 | 外部 API、文件、重试 |
| 人工复核交付 | 1,500+ | 人工时间、责任成本 |

### 6.3 风控规则

- 免费和体验版不开放高成本工具。
- 每个长任务执行前展示预计 credits。
- 单任务设置最大 credits、最大运行时间、最大重试次数和最大工具调用数。
- credits 使用量达到 50%、80%、100% 时提醒。
- 超额 credits 单价高于套餐内 credits。
- 企业客户可设置部门级预算、审批和审计。

## 8. 结果型项目包

订阅解决持续使用，项目包负责捕获一次性高价值交付。

| 项目包 | 建议价格 | 内容 | 适合场景 |
|---|---:|---|---|
| 专业交付包 | ¥1,999 / 项目 | 深度访谈、质量报告、方案页、PPT | 售前、产品立项、业务复盘 |
| 售前/咨询加速包 | ¥4,999-9,999 / 项目 | 专业交付包 + 人工复核 + 一次改稿 | 对外提案、客户汇报 |
| 企业试点包 | ¥29,800-59,800 / 季度 | 团队账号、专属场景、若干交付包、上线支持 | 部门试点、采购前验证 |

## 9. 上线节奏

### Phase 1：商业授权最小上线

- 上线体验版、标准版、专业版价格页。
- 保留当前 License 等级，补齐购买入口文案。
- 先不做复杂账单系统，可先使用管理员发码 + 人工收款。
- 所有高成本任务先用静态 credits 估算。

### Phase 2：Credits 账本

必须新增内部账本字段：

| 字段 | 说明 |
|---|---|
| `user_id` / `workspace_id` | 归属主体 |
| `task_id` | 任务唯一标识 |
| `task_type` | 访谈、报告、方案页、演示稿等 |
| `model_cost` | 模型成本 |
| `tool_cost` | 搜索、文件、外部 API 等成本 |
| `runtime_cost` | 队列、执行、存储等成本 |
| `retries` | 重试次数 |
| `credits_charged` | 实际扣费 |
| `success` / `failure_reason` | 结果和失败原因 |
| `gross_margin_estimate` | 估算毛利 |

### Phase 3：企业化销售

- 推出团队版和企业版合同。
- 增加企业级预算、审计、审批、SSO、专属实例。
- 针对咨询、售前、业务诊断行业做场景模板包。
- 建立人工复核交付流程，不把高责任结果伪装成纯自动化。

## 10. 风险与验证指标

### 10.1 主要风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 报告链路成功率不足 | 专业版毛利被重试吃掉 | 先限制 `quality` 用量，建立失败分类和补偿规则 |
| 演示稿外部服务成本不稳定 | 专业版成本失控 | 单独扣 credits，执行前提示预计消耗 |
| 用户把 Intus 当无限生成工具 | 低价套餐亏损 | 禁止 `unlimited`，保留限额和超额单价 |
| 企业客户要求强责任交付 | 自助价格无法覆盖服务成本 | 企业版和人工复核单独报价 |
| 市场锚点过低 | 价值被低估 | 把价格锚在报告/方案交付，而不是聊天工具 |

### 10.2 必须验证的指标

| 指标 | 目标 |
|---|---|
| 标准项目有效完成成本 | 稳定低于 ¥40 |
| 专业交付包有效完成成本 | 稳定低于 ¥180 |
| 标准项目成功率 | 大于 90% |
| 专业报告成功率 | 大于 85% 后再扩大自助售卖 |
| 专业版月活用户 credits 使用分布 | P95 不应突破套餐成本地板 |
| credits 毛利 | 自助套餐整体大于 70% |
| 交付包毛利 | 大于 60% |
| 标准版到专业版转化率 | 验证方案页、附录、演示稿是否是有效付费墙 |

外部基准只用于校准价格带，不作为 Intus 的直接锚点。上线前需要重新核验公开页面。

## 12. 最终建议

首期不要把价格压到普通 AI 聊天工具区间。建议以 `¥399 / 人 / 月` 的标准版作为正式交付入口，以 `¥1,299 / 人 / 月` 的专业版承接方案页和演示稿价值，再用 `¥1,999+ / 项目` 的专业交付包捕获售前、咨询和业务诊断场景的高价值需求。

最重要的工程前置条件是 credits 账本。没有可观测的任务成本、重试次数、成功率和毛利，就不要上线 `unlimited` 或大额年包承诺。
